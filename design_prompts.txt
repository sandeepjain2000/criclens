1. Ui Unified design system evaluation
Design evaluation/assessment prompts

Objective: Check if the entire UI follows one consistent design system across all screens and components, but do not provide a solution redesign.

Prompt 1:
Unified Design System Evaluation
Act as a senior UI/UX design auditor.
Evaluate the wireframe only from a system consistency perspective.
Do not redesign anything (but do not provide a solution redesign).
Analyse this codebase to validate my current wireframe against the above system consistency criteria and give me output in the following attached HTML format. The results must strictly reflect only the current wireframe.
Focus only on:
- Consistency of components (buttons, forms, tables)
- Consistent styles (border radius, shadows, icons)
- Interaction states (hover, active, disabled)
- Reuse of patterns across screens
Identify:
- System-level inconsistencies
- Pattern mismatches
- Missing or broken design rules
Do not comment on layout, usability, or content.
Output Format (Strict):
Structured table with:
| # | Category | Sub-Category | Rating (1–10) | Key Issue | Impact | Area of Action | Real-World Example | URL | Critical (Yes/No) |
Rules:
Each major section must start with a section header row:
--- SECTION NAME ---
No design system docs unless unavoidable
Keep findings specific, not generic

3. Ui Component Audit
Objective: Check if individual components are built and used correctly. but do not provide a solution redesign.

Prompt 3:
Act as a Senior UI Component Auditor.
Evaluate the wireframe strictly at a component level. Do not redesign (but do not provide a solution redesign).
Analyse this codebase to validate my current wireframe against the above system consistency criteria and give me output in the following attached HTML format. The results must strictly reflect only the current wireframe.
Evaluation Scope
Buttons
- Size, padding, and alignment consistency
- Variant usage (primary, secondary, ghost, etc.)
- Consistency across similar actions
- Disabled / loading states
Form Elements
- Inputs, dropdowns, checkboxes, radio buttons
- Label alignment and spacing
- Error states and validation behavior
Cards & Containers
- Header, body, footer consistency
- Internal spacing and structure
- Reusability of patterns
Component States
- Hover, focus, active, disabled
- Missing or inconsistent states
What to Identify
- Incorrect component usage
- Missing states
- Inconsistent patterns
- One-off or custom components breaking system
Strict Boundaries
- Do NOT evaluate layout structure
- Do NOT evaluate UX flows
- Do NOT evaluate typography system broadly
Output Structure (Strict)
Sections in Table:
- Components—Buttons
- Components—Forms
- Components—Containers
- Components—States

5. Ux Comprehensive UX Audit
Objective: Check how easy and smooth it is for users to complete tasks
Prompt 5:
Act as a UX Auditor.
Evaluate usability using real user flows. Do not redesign (but do not provide a solution redesign).
Analyse this codebase to validate my current wireframe against the above system consistency criteria and give me output in the following attached HTML format. The results must strictly reflect only the current wireframe.
Evaluation Scope
Navigation
- Clarity of movement across screens
- Predictability of actions
- Visibility of options
User Flows
- Step-by-step task completion
- Number of steps
- Friction points
Feedback & System Visibility
- Loading states
- Success / error feedback
- System response clarity
Error Handling
- Prevention
- Recovery
- Clarity of messages
Cognitive Load
- Too many decisions
- Confusing actions
- Information overload
What to Identify
- Confusing flows
- Missing feedback
- Task friction
- Dead ends
- Unclear actions
Strict Boundaries
- Do NOT evaluate visual design quality
- Do NOT evaluate layout spacing
- Do NOT evaluate component styling
Output Structure (Strict)
Sections in Table:
- UX — Navigation
- UX — Task Flows
- UX — Feedback
- UX — Errors
- UX — Cognitive Load
Additional Instructions
- Ratings must be justified in the “Key Findings” column.
- “Areas of Action” should describe what needs improvement, not how to redesign it.
- Real-world examples must be:
  - Live websites or real digital interfaces (not design system documentation).
  - Specifically relevant to school/education platforms.
  - Include direct clickable URLs only (no placeholders).
- Keep tone professional and analytical.
- Avoid speculative assumptions beyond what is visible in the wireframe.
- Highlight the Top 3 Critical Issues at the end in a short summary section.
- If evaluating multiple wireframes, include:
  - A comparative score summary table at the end.
  - An overall performance rating for each prototype.

6. Ux & Design System Evaluation Audit
Objective: Check how well UX and design consistency work together across the entire product, but do not provide a solution redesign.

Act as a Product Design Auditor.
Evaluate system-level experience across multiple screens. Do not redesign (but do not provide a solution redesign).
Analyse this codebase to validate my current wireframe against the above system consistency criteria and give me output in the following attached HTML format. The results must strictly reflect only the current wireframe.

- Scalability issues
- UX and UI misalignment
Strict Boundaries

- Do NOT analyze spacing/layout details
- Do NOT audit micro UI issues
Output Structure (Strict)
Sections in Table:
- System — Cross-Screen Consistency
- System — Patterns
- System — Scalability
- System — Accessibility
- System — UX/UI Alignment


7. Ux Labels, wording, typography, readability,
Objective: Assess wireframe labels, wording, typography, readability, consistency, and grammar quality.

Prompt 7:

Evaluate the entire wireframe focusing on labels, wording, typography, and text quality across the interface (but do not provide a solution redesign).
Analyse this codebase to validate my current wireframe against the above system consistency criteria and give me output in the following attached HTML format. The results must strictly reflect only the current wireframe.
Assess the following:
Label clarity
- Are field names, section titles, and table headers clear and understandable?
Terminology consistency
- Are similar terms and actions used consistently across the interface?
Typography hierarchy
- Do headings, labels, and table headers follow a clear visual hierarchy?
Readability
- Are font sizes, spacing, and text density easy to read and scan?
Button and action labels
- Do buttons clearly communicate the intended action?
Grammar and spelling
- Identify any typos, incorrect capitalization, or grammatical issues.
Provide the results in a table format with:
| Category | Element / Label | Issue | Severity (Low / Medium / High) | Recommendation |
Keep the evaluation strictly analytical and concise, without redesigning the interface.
Gemini: WF correction prompt
Wireframe correction prompt:

Analyze the attached report and evaluate whether any structural, layout, hierarchy, or UI consistency updates are required in the current wireframe. If improvements are necessary, implement the revisions directly within the wireframe. Do not introduce or modify the existing color palette under any circumstances.
